## 🔴 Critical (directly degraded output quality)

### B1 — PubMed EFetch returned 0 abstracts (JSON vs XML bug)
- **When:** `02:24:36`
- **Log:** `[1.4] EFetch batch 0 failed: Expecting value: line 1 column 1 (char 0)` → `Fetched 0 abstracts from PubMed`
- **Cause:** code calls efetch with `retmode=json`, but PubMed efetch only returns **XML** for `rettype=abstract`. `resp.json()` blows up → primary literature source yields nothing.
- **Impact:** lost the biggest abstract source (~30 expected → 0). Whole literature layer fell back to Europe PMC only.
- **Fix hint:** parse efetch as XML (or use esummary/efetch XML + xml.etree), don't request JSON.

### B2 — Local LLM produced unparseable JSON on 3 of 4 extraction chunks
- **When:** `02:29:06` (chunk 0), `02:32:14` (chunk 1), `02:35:24` (chunk 2) — only chunk 3 succeeded `02:38:15`
- **Log:** `[map-reduce] phase1_literature_extraction chunk N — parse failed, skipping`
- **Cause:** qwen3-4b-thinking can't reliably emit the strict nested JSON schema (records[] with nested evidence[]) for 8 dense abstracts at once.
- **Impact:** literature extraction yielded **1 gene (SPRY3)** instead of dozens.
- **Fix hint:** flatten the extraction schema, reduce chunk size for local models, add stronger re-prompt/repair loop, or use grammar-constrained / JSON-mode decoding in LM Studio.

### B3 — Local GWAS parse failed on every chunk (pandas indexing bug)
- **When:** `02:38:16`, `02:38:17`, `02:38:18`, `02:38:19`
- **Log:** `[1.5] Local GWAS parse failed: Unalignable boolean Series provided as indexer (index of the boolean Series and of the indexed object do not match).`
- **Cause:** boolean mask built across chunked `pd.read_csv` has a misaligned index vs the chunk DataFrame.
- **Impact:** GWAS genetic evidence = 0 genes (the 669MB local TSVs were effectively unused). Genetic score leaned only on OMIM.
- **Fix hint:** reset_index on each chunk before masking, or build the mask with `.reset_index(drop=True)`.

## 🟡 Performance / reliability (slow, not wrong)

### B7 — PPI centrality is the wall-clock bottleneck
- **When:** `02:38:34` build start → `02:39:16` graph built (29,403 nodes / 322,997 edges) → still computing centrality 18+ min later
- **Cause:** `closeness_centrality` (and exact-ish betweenness) on a 29k-node graph is O(V·E); dominates total runtime.
- **Impact:** single longest step in the run by far.
- **Fix hint:** subgraph to candidate genes + neighbors before centrality, cap betweenness pivots lower, skip closeness or approximate it, cache the graph.

### B8 — Per local-LLM call latency ~60–90s
- **When:** throughout (e.g. 1.1 disambiguation `02:21:10`→`02:23:43`, ~2.5 min for one gate incl. a parse-retry)
- **Cause:** qwen3-4b-thinking emits long `<think>` blocks on a 6GB GPU (~2.3GB free).
- **Impact:** ~38 calls × ~70s ≈ throughput ceiling; full 500-abstract run would be ~hours.
- **Fix hint:** batch the per-abstract prefilter (8/call) or skip for local; smaller/faster local model; cloud provider for big runs.

### B9 — EFO disambiguation needed a parse-retry (2 LLM calls for 1 gate)
- **When:** `02:22:36` + `02:23:43` (two completions for the single 1.1 gate)
- **Cause:** first structured-output attempt failed validation → LMStudio adapter re-prompted once.
- **Impact:** minor — doubled latency on the gate; succeeded on retry.
- **Fix hint:** same as B2 (local JSON reliability).

## 🔴 BLOCKER (found on 2nd run — prevents any target persistence)

### B12 — `targets` upsert fails: no unique constraint for ON CONFLICT
- **When:** end of run, final write step — `runner.py:212 → run_state.upsert_target`
- **Log:** `APIError: there is no unique or exclusion constraint matching the ON CONFLICT specification (42P10)`
- **Cause:** `upsert_target` uses `.upsert(..., on_conflict="run_id,ensembl_id")`, but `schema.sql` `targets` table has only `id primary key` — the `(run_id, ensembl_id)` unique constraint was never created.
- **Impact:** **the entire Phase 1 computation completed** (EFO, OT, literature, genetics, PPI, pathways, scoring all ran) but crashed on the first target write → **0 targets persisted, output_json never written** (mark_phase_completed runs after the loop). Ranked list was in-memory only and lost. This blocks Phase 2 too (it reads `targets` rows).
- **Subtlety:** literature-only / GWAS-forced targets have `ensembl_id=""`; a unique on `(run_id, ensembl_id)` would collide across those empty IDs. **Phase 2 matches targets by `(run_id, symbol)`** → the correct unique key is `(run_id, symbol)`, not ensembl_id.
- **Fix hint:** add `unique(run_id, symbol)` to `targets` in schema.sql **and** change `upsert_target` `on_conflict="run_id,symbol"`. Coordinate — `run_state.py` is shared with the Phase 2/3 work.
